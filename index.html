How it works

01

Point it at your project

Provide a local path to your codebase and log folder. Bakup recursively scans source files and log directories in read-only mode — tagging each entry by severity. Nothing is indexed remotely. Nothing leaves disk.

$ bakup index --path /your/project --logs /your/project/logs
02

Ask what happened

Type a question in plain English. Bakup searches across your code and all indexed log files — correlating errors across multiple sources to surface the most relevant matches.

$ bakup ask "Have we seen this null pointer before?"
03

Get a structured summary with citations

Responses include a structured summary referencing the specific files, functions, and log lines that informed the answer — with cross-file error distribution and a multi-factor confidence score. You decide what to act on.

> auth/session.js:142 — confidence: 0.87
04

Copy the answer into your postmortem

Every response includes the answer, confidence score, source file, and line numbers. Copy the JSON or the UI output directly into your incident timeline or runbook — sources intact.

> confidence: 0.84 · src/auth/session.py:28–45 · log:412

What the engine does

Recursive log folder ingestion

Point Bakup at a directory. It walks every subfolder, finds .log and .out files, and indexes each entry individually.

Cross-file error aggregation

Errors are correlated across multiple log files. The response tells you which files are affected, how errors are distributed, and where the dominant failure source is.

Severity-aware analysis

Each log entry is automatically tagged as error, warning, or info at ingestion time. Retrieval and scoring prioritize higher-severity results.

Structured summaries with confidence

Answers include a multi-factor confidence score weighing similarity, error volume, severity, recency, pattern consistency, and cross-file corroboration.

Your code never leaves your machine.
That's not a setting. That's the architecture.

Read-only filesystem access

Bakup requests read permissions only. It cannot create, overwrite, or delete files. There is no write path in the codebase.

No network calls during analysis

Once installed, Bakup makes zero outbound connections when processing your project. Analysis runs entirely on your hardware. Verify this with a network monitor while it runs.

No telemetry. No crash reports.

We do not collect information about how you use Bakup, what projects you point it at, or what questions you ask. We have no pipeline to receive it.

Today, Bakup runs entirely on your machine. A managed cloud mode is on the roadmap — but local-first will always remain the default. The tool has no server-side dependency.

Docker. Four steps. Under ten minutes.

  1. 1

    Unzip, copy the env file, and set your project path. Docker handles everything else.

    # Unzip the archive
unzip bakup-preview-0.1.0.zip && cd bakup
# Configure — set BAKUP_PROJECT_PATH to your project directory
cp .env.example .env▌
  2. 2

    Start the backend and local UI. First boot downloads the embedding model (~90 MB).

    docker compose up
  3. 3

    Index your project and log folder. Bakup scans recursively and tags log entries by severity. Returns a namespace you use to query.

    curl -s -X POST http://localhost:8000/index \
  -H "Content-Type: application/json" \
  -d '{"path":"/your/project","log_path":"/your/project/logs"}'
  4. 4

    Ask a question. Paste the namespace from step 3.

    curl -s -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"Have we seen connection timeouts before?","namespace":"<from step 3>"}'

Privacy

No data leaves your machine. Bakup processes your codebase and logs entirely on local hardware. There are no outbound network calls during analysis, no telemetry, and no crash reporting.

No tracking. We do not collect information about what projects you analyze, what questions you ask, or how you use the tool. There is no analytics pipeline to receive such data.

Access keys are used solely to gate distribution of the developer preview. They are validated client-side via SHA-256 hashing. We do not log key usage or tie keys to analytics.

Read-only access. Bakup requests read permissions only. It cannot create, modify, or delete files on your filesystem.

Future cloud mode. A managed cloud option is on the roadmap for team collaboration. It will operate under a separate privacy policy with explicit opt-in. Local-first will remain the default.
