# bakup.ai — Project Context

Local-first AI incident intelligence tool. Indexes codebases and log files,
then answers natural-language questions about errors, patterns, and root causes.

---

## Installer Workflow

```
User downloads bakup-ai-installer.exe  (from landing page or /download route)
  │
  ▼
Run installer  (Inno Setup, LZMA2 compressed, ~240 MB)
  │
  ├─ Extracts compiled binary to:
  │    C:\Users\<user>\AppData\Local\Programs\bakup-ai\
  │
  ├─ Installs files:
  │    bakup-launcher.bat            — entry point script
  │    bakup-server\bakup-server.exe — compiled Python backend
  │    bakup-server\ui\              — static HTML/CSS/JS
  │    bakup-server\backend\         — bundled backend modules
  │    bakup-server\_internal\       — Python runtime + wheel data
  │    bakup-server\*.dll / *.pyd    — native extensions (torch, chromadb)
  │
  ├─ Creates directories:
  │    data\vectordb\                — ChromaDB persistent storage
  │    data\model-weights\           — embedding model cache
  │
  ├─ Creates shortcuts:
  │    Start menu → bakup.ai
  │    Desktop    → bakup.ai  (checked by default)
  │
  └─ Post-install: launches bakup-launcher.bat (optional, user can skip)
```

## Launch Sequence (bakup-server.exe)

```
bakup-server.exe starts
  │
  ├─ 1. Path setup
  │    ├─ Detect frozen state (sys.frozen)
  │    ├─ Resolve APP_DIR (exe location)
  │    ├─ Create DATA_DIR alongside exe
  │    └─ Set env defaults (access key, chroma dir, model cache)
  │
  ├─ 2. Module resolution
  │    ├─ Add backend/ from sys._MEIPASS to sys.path
  │    └─ Run access key check (exits if wrong key)
  │
  ├─ 3. Port auto-detection  (core/net.py)
  │    ├─ Try to bind preferred port (default 8000)
  │    ├─ If busy, scan 8001 → 8019 for next free port
  │    ├─ Update settings.port and BAKUP_PORT env var
  │    └─ Log: "bakup: port 8000 is in use — falling back to 8001"
  │
  ├─ 4. FastAPI app construction
  │    ├─ CORS middleware (localhost origins, includes resolved port)
  │    ├─ Routes: /health, /index, /ask, /ask/stream, /llm/*, /debug/*
  │    └─ Mount UI static files from bakup-server/ui/ (html=True)
  │
  ├─ 5. Lifespan startup
  │    ├─ ensure_models_downloaded()  — downloads embedding model on first run (~90 MB)
  │    ├─ init_vector_store()         — initializes ChromaDB
  │    └─ Prints "bakup: ready."
  │
  ├─ 6. Browser auto-open  (background thread)
  │    ├─ Polls GET /health every 1 second on the resolved port
  │    ├─ Waits up to 120 seconds (first-run model download)
  │    ├─ Opens http://127.0.0.1:<resolved_port> when health returns 200
  │    └─ Fallback: opens browser after 120s even if not ready
  │
  └─ 7. Uvicorn server
       └─ Listens on 127.0.0.1:<resolved_port>, blocking
```

## Port Detection (core/net.py)

```
resolve_port(host, preferred_port)
  │
  ├─ Try socket.bind(host, preferred_port)
  │    └─ No SO_REUSEADDR (Windows would allow port stealing)
  │
  ├─ If bind succeeds → return preferred_port (no change)
  │
  └─ If bind fails (OSError) → scan upward
       ├─ Try preferred_port + 1, +2, ... +19
       ├─ Return first port that binds successfully
       ├─ Log: "bakup: port N is in use — falling back to M"
       └─ RuntimeError if all 20 candidates are busy
```

**Applies to both entry points:**
- `backend/main.py` — development (resolves port before app construction)
- `build/bakup_server.py` — compiled binary (same logic)

**Why before app construction:** The CORS `allow_origins` list includes
`http://localhost:<port>` and `http://127.0.0.1:<port>`. If the port changes
after CORS is configured, the browser UI would be blocked by CORS.

## Code-Aware Ingestion

### Supported Languages

| Language | Extensions | Parser Strategy |
|----------|-----------|-----------------|
| Python | `.py` | Regex + indentation-based block end |
| JavaScript | `.js`, `.jsx` | Regex + brace counting |
| TypeScript | `.ts`, `.tsx` | Regex + brace counting |
| Go | `.go` | Regex + brace counting |
| Java | `.java` | Regex + brace counting |
| JSON | `.json` | `json.loads()` + regex key locations |
| YAML | `.yaml`, `.yml` | Section-based splitting |
| TOML | `.toml` | Section-based splitting |
| INI/Config | `.ini`, `.cfg`, `.conf`, `.env` | Section-based splitting |
| Other | everything else | Naive 40-line window (fallback) |

No external AST dependencies — all parsing uses stdlib `re` and `json`.

### Ingestion Pipeline

```
POST /index  { project_path: "..." }
  │
  ├─ walk_project(project_path, namespace=namespace)
  │    │
  │    ├─ For each file:
  │    │    ├─ detect_language(filepath)  →  "python" | "javascript" | ...
  │    │    │
  │    │    ├─ If supported language:
  │    │    │    ├─ chunk_file_code_aware(filepath, project_root, namespace=namespace)
  │    │    │    │    ├─ parse_file(text, language)  →  List[CodeUnit]
  │    │    │    │    ├─ code_units_to_chunks(units)  →  List[Chunk]
  │    │    │    │    └─ build_graph_from_units(units, file)  →  populates symbol graph
  │    │    │    └─ Each chunk gets: context header, metadata fields
  │    │    │
  │    │    └─ If unsupported language:
  │    │         └─ chunk_file(filepath, project_root)  →  naive 40-line chunks
  │    │
  │    └─ Returns all Chunk objects
  │
  ├─ embed_texts([chunk.text for chunk in chunks])
  ├─ add_chunks(chunks, embeddings)  →  ChromaDB (with code metadata)
  │
  └─ Post-indexing:
       ├─ build_architecture_summary()  →  cached per namespace
       └─ Log: symbol graph stats (nodes, edges), architecture stats (modules, entry points)
```

### Code Parsing Strategy

Each language parser extracts `CodeUnit` objects:

| Parser | Detects | Block End Strategy |
|--------|---------|-------------------|
| Python | classes, methods, standalone functions, imports | Indentation: stops when indent ≤ base indent |
| JavaScript/TypeScript | classes, functions (named + arrow), imports | Brace counting: tracks `{` / `}` depth |
| Go | funcs (including receiver methods), structs | Brace counting |
| Java | classes, methods | Brace counting |
| JSON | top-level keys as separate blocks | `json.loads()` + regex for key line positions |
| YAML/TOML/INI | sections (top-level keys, `[tables]`, `[sections]`) | Next section start or EOF |

Python-specific: collects decorators (`@`), triple-quote docstrings, leading comments.
JS/TS-specific: collects JSDoc (`/** ... */`), handles `export`, `async`, arrow functions.
Go-specific: collects C-style comments, handles receiver methods `func (s *Server) Method()`.

### Chunking Rules

| Rule | Value |
|------|-------|
| Max unit size | 80 lines — larger units are split with overlap |
| Split overlap | 8 lines — ensures context continuity across splits |
| Min merge threshold | 10 lines — small adjacent units not merged (each stays independent) |
| Context header | Prepended to each chunk: `# File:`, `# Language:`, `# Class:`, imports |

Context header example:
```
# File: src/auth/session.py
# Language: python
# Class: SessionStore
import os
from datetime import datetime
```

Unclaimed lines (module-level code, comments, constants) are collected into a
`module` unit — ensuring no source code is dropped during parsing.

### Metadata Schema

Every `Chunk` object carries these fields into ChromaDB:

| Field | Type | Example | Stored In |
|-------|------|---------|-----------|
| `source_file` | str | `src/auth/session.py` | ChromaDB metadata |
| `language` | str | `python` | ChromaDB metadata |
| `function_name` | str | `create_session` | ChromaDB metadata |
| `class_name` | str | `SessionStore` | ChromaDB metadata |
| `chunk_kind` | str | `function` / `class` / `method` / `module` / `config_block` | ChromaDB metadata |
| `docstring` | str | `Create a new session...` (truncated to 500 chars) | ChromaDB metadata |
| `imports` | str | `import os\nfrom datetime import datetime` (truncated to 500 chars) | ChromaDB metadata |
| `start_line` | int | `18` | ChromaDB metadata |
| `end_line` | int | `27` | ChromaDB metadata |

All fields default to empty string — backward compatible with chunks from the
naive text splitter.

### Retrieval Behavior

Code metadata flows through the full retrieval pipeline:

```
query_chunks()  →  RetrievedChunk (with metadata)
  │
  └─  rank_results()  →  RankedResult (with confidence boosting)
       ├─  +0.05 confidence for chunk_kind in {function, method, class}
       └─  +0.02 confidence for non-empty docstring
```

This ensures that structurally meaningful code (functions, classes) ranks higher
than module-level leftovers when relevance scores are close.

### Debug Logging

The `bakup.ingestion` logger emits at `DEBUG` level during indexing:

```
Code-aware ingestion complete: 42 file(s), 187 chunk(s) [28 function(s), 5 class(es), 31 method(s)]
Chunks per file (top 20): {'src/auth/session.py': 7, 'src/api/routes.py': 12, ...}
```

Per-file parsing is logged by `bakup.code_chunker`:
```
Parsed src/auth/session.py (python): 7 unit(s) → 7 chunk(s) [0 func, 1 class, 5 method]
```

## Symbol Graph (`symbol_graph.py`)

In-memory per-namespace graph built during ingestion. Tracks:
- `function → calls` (which functions it calls)
- `class → methods` (its own methods)
- `module → imports` (what it depends on)
- `file → defines` (functions/classes in it)

### Build

Built automatically during `chunk_file_code_aware()` when a namespace is
provided. Processes `CodeUnit` objects to populate nodes + edges.

```
walk_project(path, namespace="my-project")
  └─ chunk_file_code_aware(file, root, namespace="my-project")
       └─ build_graph_from_units(units, file_path, namespace)
            ├─ Adds SymbolNode per function/class/method
            ├─ Adds "defines" edges (file → symbol)
            ├─ Adds "imports" edges (file → imported module)
            ├─ Extracts calls via regex: \b([a-zA-Z_]\w*)\s*\(
            └─ Adds "calls" edges (function → called function)
```

### Query

Structural questions answered without LLM or embedding:

| Pattern | Example | Method |
|---------|---------|--------|
| `which files use/import X?` | "which files import auth?" | `files_that_import()` |
| `what depends on X?` | "what depends on database?" | `dependents_of()` |
| `what methods does X have?` | "what methods does UserService have?" | `methods_of_class()` |
| `what functions are in X?` | "what functions are in routes.py?" | `symbols_in_file()` |

### Storage

Singleton dict: `_graphs: Dict[str, SymbolGraph]`. Accessed via `get_graph(namespace)`.
Survives across requests within the same server process.

### Debug Endpoint

`GET /debug/symbols/{namespace}` — returns node/edge counts, top files by symbol count,
top imported modules.

## Context Bundler (`context_bundler.py`)

Prevents sending fragmented code to the LLM by grouping related chunks.

### Bundle Structure

```
ContextBundle
  ├─ primary    : RankedResult   (highest-confidence match)
  ├─ siblings   : List[RankedResult]  (adjacent chunks from same file, ≤100 lines gap)
  └─ import_chunks : List[RankedResult]  (chunks defining imported symbols)
```

### Integration

Activated in `rag.py` for non-log queries with code results:

```
ranked results
  │
  └─ bundle_context(ranked, top_n=5)
       ├─ For each top result:
       │    ├─ Find siblings (same file, within 100 lines)
       │    └─ Find import definitions (functions/classes matching imported names)
       └─ bundles_to_ranked_list(bundles)  →  enriched ranked list for LLM
```

### Parameters

| Parameter | Default | Effect |
|-----------|---------|--------|
| `top_n` | 5 | Number of primary results to bundle |
| `max_siblings` | 2 | Max adjacent chunks per primary |

## Architecture Summarizer (`architecture.py`)

Generates a high-level project overview after indexing.

### What It Detects

| Category | Method |
|----------|--------|
| Entry points | Regex match: `main.py`, `app.py`, `server.py`, `index.js`, `manage.py`, `wsgi.py` |
| Config files | Regex match: `Dockerfile`, `package.json`, `requirements.txt`, YAML/TOML/INI |
| Module purposes | Directory name mapping: `api/` → "API handlers", `models/` → "Data models" |
| Languages | File extension counting |
| Core dependencies | Most-imported modules from symbol graph |

### Output

Rendered as markdown via `summary_text()`:
- Languages (with file counts)
- Project overview (files/functions/classes)
- Entry points
- Modules/Services (with descriptions)
- Core dependencies (import counts)
- Config files
- ASCII directory tree (max depth 3)

### Integration

- Built during `_run_local_ingestion()` in `index.py` after chunks are stored
- Served directly for architecture queries (`_is_architecture_query()` in `rag.py`)
- Debug: `GET /debug/architecture/{namespace}`

### Storage

Singleton dict: `_summaries: Dict[str, ArchitectureSummary]`.

## Log-to-Code Cross Analysis (`log_code_linker.py`)

Links log errors to indexed source code for root-cause analysis.

### Reference Extraction

7 regex patterns scan log text for code references:

| Pattern | Example | Confidence |
|---------|---------|------------|
| Python traceback | `File "auth.py", line 42, in login` | 0.95 |
| Java stack trace | `at com.app.Service.method(Service.java:42)` | 0.90 |
| JS/Node stack trace | `at handleRequest (server.js:42:10)` | 0.90 |
| Go panic | `auth/session.go:42` | 0.90 |
| Generic file:line | `session.py:42` | 0.80 |
| Class.method | `UserService.authenticate` | 0.70 |
| Function reference | `create_session()` | 0.50 |

### Cross-Analysis Flow

```
Log query detected  +  code chunks available
  │
  ├─ extract_code_references(log_text)      → CodeReference list
  ├─ link_logs_to_code(log_chunks, code_chunks)
  │    ├─ Build indices: file_path → code, function_name → code, class_name → code
  │    └─ Match references against indices  → LogCodeLink list
  └─ build_cross_analysis_context(links)    → formatted context block
       └─ Appended to log analysis context for LLM
```

### LLM Integration

When cross-analysis finds links, `rag.py` uses `SYSTEM_CROSS_ANALYSIS` prompt
instead of `SYSTEM_LOG_SUMMARY`. This prompt instructs the LLM to:
- Map each error to its source code
- Explain root cause by reading the actual code
- Trace call chains across functions
- Suggest concrete fixes

## Download Route (`/download`)

```
GET /download
  │
  ├─ Resolves: backend/api/routes/download.py → project root → downloads/
  │            _DOWNLOAD_DIR = bakup.ai/downloads/
  │            _INSTALLER_NAME = bakup-ai-installer.exe
  │
  ├─ If file missing → 404 JSON { "error": "..." }
  │
  └─ If file exists → FileResponse (streaming, not loaded into memory)
       ├─ Content-Type:            application/octet-stream
       ├─ Content-Disposition:     attachment; filename="bakup-ai-installer.exe"
       ├─ Content-Length:          <file size in bytes>
       ├─ X-Content-Type-Options:  nosniff
       └─ Cache-Control:           no-cache
```

**HEAD support:** Same route handles HEAD (CDN probes, link checkers).

**Scope:** Registered in `backend/main.py` (dev) only. NOT in `build/bakup_server.py`
(compiled binary) — the installed app should not serve itself for download.

## Hosting Requirements

| Requirement | Detail |
|-------------|--------|
| Static file hosting | Landing page (`index.html`, `styles.css`, `app.js`, `fonts.css`) |
| Binary file hosting | `downloads/bakup-ai-installer.exe` (~240 MB) — same origin as landing page |
| HTTPS | Recommended — browsers may block `.exe` downloads over plain HTTP |
| CORS | Not needed for the landing page (same-origin static files) |
| `.exe` MIME type | Host must serve `.exe` as `application/octet-stream` (most do by default) |
| Max upload size | Hosting provider must allow ≥250 MB file uploads |
| Bandwidth | ~240 MB per download — plan accordingly for preview distribution |

**GitHub Pages:** Works for landing page + installer if repo size allows.
Alternative: host the installer on GitHub Releases and update the `href` in `index.html`.

**Self-hosted:** Run `backend/main.py` with the landing page served via a reverse proxy
or separate static server. The `/download` route handles binary delivery.

## PyInstaller Build Configuration

| Setting | Value | Rationale |
|---------|-------|-----------|
| Mode | Folder distribution (COLLECT) | One-file mode extracts to temp on every launch — slower, more AV flags |
| UPX | **Disabled** (`upx=False`) | UPX-packed binaries mimic malware packers — #1 cause of AV false positives |
| Version info | `build/version_info.txt` | Embeds CompanyName, FileDescription, ProductName — reduces AV "unknown publisher" flags |
| Console | `True` | Users see server logs in terminal window |
| Debug | `False` | No debug bootloader |
| noarchive | `False` | .pyc files bundled in archive (smaller, faster) |
| Spec file | `build/bakup.spec` | Defines hidden imports, data bundling, excludes |
| Entry point | `build/bakup_server.py` | Direct `uvicorn.run(app)` — no string import |

### Excluded modules (from spec)
- GUI: tkinter, matplotlib, PIL, IPython, jupyter
- Test: pytest, unittest, doctest
- Torch GPU: torch.cuda, torch.xpu, torch.mps, torch.mtia
- Torch dev: torch.testing, torch.distributed, torch._inductor, torch._dynamo, torch.onnx
- Dev tools: setuptools, pip, pdb, cProfile, lib2to3
- Unused: tensorboard

### Post-build cleanup (from build.ps1)
Removes directories that leak through `collect_all` despite excludes:
- `backend/.pytest_cache`, `chromadb/test`
- `torch/testing`, `torch/distributed`, `torch/_inductor`, `torch/_dynamo`
- `torch/onnx`, `torch/_export`, `torch/profiler`, `torch/package`, `torch/compiler`

### Expected binary sizes
| Artifact | Size |
|----------|------|
| `bakup-server.exe` | ~44 MB |
| `dist/bakup-server/` (total) | ~780 MB (after cleanup) |
| `bakup-ai-installer.exe` | ~240 MB (LZMA2 compressed) |

## Security Considerations

- **No UPX packing** — eliminates the most common AV false positive trigger
- **Version info embedded** — binary has proper publisher metadata (CompanyName, FileDescription)
- **Console mode** — no hidden background process; user sees server activity
- **Localhost only** — server binds to `127.0.0.1`, not `0.0.0.0`
- **No CUDA/GPU binaries** — only CPU inference; no large GPU DLLs bundled
- **Access key required** — server exits immediately if `BAKUP_ACCESS_KEY` is wrong
- **Test artifacts stripped** — no pytest/unittest code in production binary
- **Code signing** — not yet implemented; recommended before public distribution
- **Icon** — not yet added (`icon=None`); a proper .ico file improves AV trust
